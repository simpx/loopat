## 1. 概念精确定义

### 1.1 Loop

**定义**：first-class 工作单元，绑定一个长程任务和一个 driver。

**结构**：
```
loop/<id>/
├─ chat/         持续对话历史（user / ai 两种角色 + driver-change 系统标记）
├─ work/         工作 dir（通常是某个 repo 的 git worktree）
├─ meta.json     id / title / driver / created / status / linked_focus[]
└─ artifacts/    输出物（diff、文档、跑通的命令等）
```

**生命周期**：

```
open ──active──→ closed
            │
            ├─ forked     (开新分支 loop)
            └─ archived   (long-term 不动)
```

**关键约束**：
- 一个 loop 同一时刻只有一个 driver（人）
- driver 可被 transfer（产生一条 driver-change 系统消息）
- chat 不显示个人名字（统一 `user` / `ai`），driver 切换通过系统标记呈现
- close 后仍可读、可 fork，但默认不再 active

**操作**：`spawn` / `fork` / `transfer-driver` / `close` / `archive`

### 1.2 Focus

**定义**：团队当下"什么重要"的 view。是注意力的物化，不是 todo list。

**结构**：每个 focus item 是一个简短标题 + 关联的 loop 集合（m:n）。

**三档**：
| 档 | 行为 | 来源 |
|---|---|---|
| **pinned** | 永不归档 | 手动 pin |
| **focus** | 8 天无活动自动归档 | 手动 add |
| **archived** | 不显示在默认视图 | 自动归档或手动 |

**活动定义**：focus 关联的任意 loop 有新一轮 chat → focus 时间刷新。
**附加显示**：底段列出"任意人 driving 但未挂 focus 的 active loop"，自动出现，不需要 add。

**约束**：
- focus 不是任务，没有 done 状态。归档 ≠ 完成
- focus 不绑定 driver
- m:n loop 关联（一个 loop 可挂多个 focus，反之亦然）

### 1.3 Context

> team's distilled materials —— 团队精炼过、可被反复使用的原料。

三种形态：

#### Knowledge（passive · markdown）

- `inbox/`：临时草稿、零散记录（高熵）
- `knowledge/`、`loop/`、`ai-org/` 等稳定主题目录：沉淀的稳态知识（低熵）
- 写法：markdown + frontmatter（`title / tags / updated / driver`）+ wikilink `[[X]]` + 自动 backlinks
- 多人共同维护，有 git revision

#### Agents（active · executable）

- 跑在任何地方（云、本机、容器、cron、k8s），通过 1001 API 接入：
  - `GET /context` 拉知识
  - `GET /focus` 看团队当下注意力
  - `SUB /chat/<channel>` / `POST /chat/<channel>` 收发消息
  - `POST /loops` 主动创 loop
- persona = system prompt + tool 集 + 触发模式
- Trigger 三种：`mention` / `schedule` / `event`
- 来源：从反复出现的 Loop pattern 里 distill 出来 —— **agent 是把"读 doc 然后做事"这套流程编码进可执行外壳**

#### Repos（passive · code）

- 团队 register 的 git 仓
- 每个 repo 显示 recent loops + spawn loop 入口
- spawn loop = 后台 `git worktree add` → 创建新 loop 的 work dir

**Loop ↔ Agent 对照**：

| | Loop | Agent |
|---|---|---|
| 谁的 | 一个 driver（人）的事 | 团队资产（无人 driver） |
| 时长 | 数小时数天，会 close | 长期运行 |
| 触发 | 人决定开 | 持续监听 / 定时 |
| 终点 | artifact → close | 没终点（除非下线） |
| 改 persona | 不（一次性的事） | 有（迭代 prompt、加 tool） |
| 数量 | 多（几十） | 少（几个，每个明确职责） |

> Loop 是火，Agent 是值班的炉子。

### 1.4 Chat

- 多 channel + 单聊 / 群聊
- bot 是一次性回应（信息查询、统计、提醒）
- 可在某条消息 `spawn loop`，把讨论升级成深加工
- loop 的进展可回流进 chat，让团队感知，不需要别人 attach

**Chat vs Loop**：

| | Chat | Loop |
|---|---|---|
| 时长 | 几条消息 | 数小时 / 数天 |
| 上下文深度 | 浅 | 深（dir + 持续 chat + artifact）|
| 例子 | "xx 命中率为啥降了" | "上线 gateway" |

### 1.5 概念关系图

```
chat ──[spawn]──→ loop ──[沉淀]──→ knowledge / focus
                    │
                    ├──→ repo (git worktree)
                    │
                    └──→ agent (反复出现的 pattern → 长期值班)
```

### 1.6 顶层组织

10 人左右共享一个 **workspace**（名字待定 —— team 更直接但 workspace 强调"空间"，可跨 team 复用）。
Workspace 是隔离边界，跨 workspace 的 loop / context / focus 不互通。

## 2. MVP 目标

### 2.1 要验证的 hypothesis

| # | Hypothesis | 验证方式 |
|---|---|---|
| H1 | 把"工作"与"chat"显式分开能让协同更顺 | 1 个 driver + 1 个 attacher 完整跑一个 loop，主观评 vs 在 Slack 里聊 |
| H2 | Loop 作为 first-class 概念能取代多个工具拼凑 | driver 在 loop 里同时完成 chat + 文件编辑 + 命令，不切走 |
| H3 | Focus 比 todo 更不焦虑 + 同样有用 | 用一周 focus，主观评 vs 用 todo / 用 Tracker |
| H4 | Context（knowledge + agent + repo）的统一让 AI 协作 footprint 更小 | agent 跑在 chat 里，能正确 distill 一段 chat 进 inbox / knowledge |
| H5 | 统一 C/S 架构能既保留 local 工具链又支持 attach 共享 | 队友能 attach 我本地 loop，看到我的 chat / 文件 / 命令实时进展 |

### 2.2 "哇" 时刻（具体可演示的 moment）

1. 进来看到 workspace + 4 个一级概念，结构一眼看懂
2. 在 chat 里聊着聊着 `spawn loop`，**当场切到 Loop tab 接着干**（不切工具、不切窗口）
3. Loop 里 AI 自动把关键决定流回 inbox，切到 Knowledge tab 看到 AI 整理过的笔记
4. 队友在他的浏览器 attach 我的 loop，**看到我和 AI 的实时 chat**（C/S 共享真起作用）
5. 切到 Focus，看到团队当下在意的事，没有"待办列表"那种焦虑

### 2.3 不验证什么（明确 out of scope）

- ❌ 多 workspace 之间的协同（先 1 个 workspace）
- ❌ 完整的权限模型（先信任所有人）
- ❌ 移动端 / 离线（先桌面 + 在线）
- ❌ 自建 LLM gateway（直接用 Claude API）
- ❌ 高级 agent 编排（agent 先跑通 mention trigger）
- ❌ 真实持久化的 wikilink backlink 索引（先 mock）
- ❌ Knowledge 的 git 多人同步冲突解决（先单写）

## 3. MVP 范围与设计

### 3.1 实现路线（基于 fork opencode）

opencode 提供：Bun + Hono 后端、SolidJS + Tailwind 前端、Tauri 2 桌面壳、SSE 多客户端、Markdown 渲染。
我们已经验证：fork 出一个 `1001-prototype` 分支，把原 opencode 的 project / session UI 隐藏，挂上我们的 4-tab shell。

**MVP 分四个阶段：**

| Phase | 内容 | 状态 |
|---|---|---|
| **P0 高保真原型** | 4-tab shell + 每 tab 的 mock 数据，用来 demo 概念和确认设计 | 进行中 |
| **P1 真实 Loop** | spawn / chat / driver-transfer / close 串通，hook 真 Claude API | 待启动 |
| **P2 真实 Focus** | focus item CRUD、loop 关联、8 天归档定时 | 待启动 |
| **P3 真实 Knowledge** | markdown 持久化、wikilink + backlinks、frontmatter 解析 | 待启动 |
| **P4 Attach 协议** | 队友 attach 我的 loop，C/S SSE 多客户端镜像 | 设计中 |

Agent / Chat 跨阶段渐进。

### 3.2 4 个 tab 的 UI 范围（MVP）

**Loop tab**（最完整）
- 3 列 IDE 风格：左 loop list / 中文件树 / 右 chat + terminal drawer
- chat 不显示个人名字，driver 切换插系统标记
- "driver 是我"的视图

**Focus tab**（zen）
- 侧栏极简：当下 / 归档
- 主区三段：📌 Pinned / focus（8d expires）/ active loops not in any focus

**Context tab**（最大）
- 顶部 horizontal sub-nav：Knowledge / Agents / Repos
- 每个 sub-tab 独占 sidebar+main
- Knowledge 子 tab 是 MVP 的重点：Obsidian-like 体验

**Chat tab**（轻量）
- Slack-like channels + DMs
- 每条消息可 `spawn loop`
- bot 一次性回应

### 3.3 关键架构选择

**约定大于配置**
- 目录结构、文件命名、frontmatter 字段、agent persona 字段、API 端点 —— 都是约定
- 用户不能配置层级、字段名、tool 集语义

**统一的 C/S 架构**
- 每个 loop 跑在某个 server 上 —— server 既可以是本机也可以是云端
- client 走同一套 attach 协议
- 共享 local server 的安全 trade-off（端口、权限、账户）是协作不可避免的妥协

## 4. 待解（按优先级）

1. **Loop 共享的安全边界**：C/S attach 协议的权限模型、本机 server 暴露端口的安全 trade-off 还没想清楚
2. **Workspace 隔离 + 权限**：账户系统、谁能看哪个 loop / focus / knowledge
3. **Agent 提炼回流的触发机制**：被动监听 / 定时跑 / on-demand 触发
4. **Knowledge 的 wikilink + backlinks 持久化索引**：原型是 mock，要有真实的反向索引和增量维护
5. **Loop 的 dir 与 repo worktree 映射**：spawn loop 时 git worktree add 的策略，close / archive 时 worktree 怎么回收
